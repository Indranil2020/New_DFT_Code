# T5.7: TIDES Scale-Out Interface Specification

**Status:** Design document (no code)
**Phase:** B (interface spec for Phase C implementation)
**Last updated:** 2026-07-09

---

## 1. Purpose

Define the interfaces for distributing TIDES linear-scaling solvers (R2/SP2,
R3/FOE) across multiple GPUs and nodes. This spec covers:

- Data partitioning (tile distribution)
- Communication primitives (halo exchange, all-reduce)
- Solver dispatch for distributed mode
- API surface for the Python layer

Target: 10⁵–10⁶ atoms on multi-GPU / multi-node systems (Phase C, M31–M48).

## 3. Data Partitioning

### 3.1 Tile Distribution

The TileMat (CSR-of-tiles) is partitioned by atom blocks using the graph
partitioner (T8.2 RCB or METIS). Each partition owns:

- **Local tiles:** Tile blocks where both row and column atoms are local.
- **Halo tiles:** Tile blocks where one atom is local and the other is remote.

```cpp
struct DistributedTileMat {
  // Partition metadata
  int rank;
  int n_partitions;
  std::vector<int> atom_ownership;  // atom_id → partition rank

  // Local tile data (CSR-of-tiles for local atoms)
  TileMat local_tiles;

  // Halo tiles (tiles crossing partition boundaries)
  struct HaloTile {
    int local_atom;
    int remote_atom;
    int remote_rank;
    TileBlock block;
  };
  std::vector<HaloTile> halo_tiles;

  // Communication plan
  HaloExchangePlan halo_plan;
};
```

### 3.2 Partitioning Interface

```cpp
class Partitioner {
 public:
  // Partition atoms into n_parts groups, minimizing edge cut.
  // Input: atom positions, tile adjacency graph, n_parts.
  // Output: atom_id → partition assignment.
  static std::vector<int> Partition(
      const std::vector<double>& positions,
      const std::vector<std::vector<int>>& adjacency,
      int n_parts,
      Method method = Method::RCB);  // RCB or METIS
};
```

### 3.3 Density Matrix Distribution

For SP2 (R2) and FOE (R3), the density matrix P is distributed:

- Each partition owns rows of P corresponding to its local atoms.
- Columns span all atoms (local + halo).
- Halo columns are updated via exchange after each SP2 iteration or FOE expansion.

## 4. Communication Primitives

### 4.1 Halo Exchange

```cpp
class DistributedHaloExchange {
 public:
  // Initialize from partition + halo plan.
  void Init(const HaloExchangePlan& plan, CommBackend backend);

  // Exchange halo tile data between partitions.
  // Blocking: returns when all halo tiles are updated.
  void Exchange(std::vector<HaloTile>& halo_tiles);

  // Non-blocking: start exchange, continue local compute, then wait.
  void StartExchange(std::vector<HaloTile>& halo_tiles);
  void WaitExchange();

  // Overlap: exchange while computing local tiles.
  void ExchangeAndCompute(
      std::vector<HaloTile>& halo_tiles,
      std::function<void()> local_compute);
};
```

### 4.2 Reductions

```cpp
class DistributedReductions {
 public:
  // All-reduce scalar (energy, trace, etc.)
  double AllReduce(double local_value, Op op = Op::Sum);

  // All-reduce vector (gradient, density, etc.)
  void AllReduce(std::vector<double>& local, Op op = Op::Sum);

  // Norm-2 across partitions
  double Norm2(const std::vector<double>& local);
};
```

### 4.3 Communication Backends

| Backend | Use Case | API |
|---------|----------|-----|
| NCCL | Single-node multi-GPU | `ncclAllReduce`, `ncclSend`/`ncclRecv` |
| NVSHMEM | Multi-node, fine-grained | `nvshmem_*` (PGAS, one-sided) |
| MPI | CPU-side orchestration | `MPI_Allreduce`, `MPI_Send`/`MPI_Recv` |

Selection: NCCL for ≤8 GPUs on one node, NVSHMEM for multi-node GPU, MPI for
CPU-only fallback.

## 5. Distributed Solver Interface

### 5.1 Distributed SP2 (R2)

```cpp
class DistributedSP2 {
 public:
  struct Config {
    double truncation_threshold;
    int max_iterations;
    bool use_gpu;  // local GPU SP2
  };

  // Run distributed SP2 purification.
  // Each partition runs local SP2 on its tile blocks,
  // with halo exchange between iterations.
  static Status Purify(
      DistributedTileMat& H,
      DistributedTileMat& P,
      double mu,         // chemical potential
      const Config& cfg,
      DistributedHaloExchange& halo,
      DistributedReductions& reductions);
};
```

### 5.2 Distributed FOE (R3)

```cpp
class DistributedFOE {
 public:
  struct Config {
    int chebyshev_order;
    double spectral_min, spectral_max;
    double temperature;
    bool use_gpu;
  };

  // Run distributed FOE expansion.
  // Each partition computes local Chebyshev moments,
  // with halo exchange for each matrix-power step.
  static Status Expand(
      DistributedTileMat& H,
      DistributedTileMat& P,
      double mu,
      const Config& cfg,
      DistributedHaloExchange& halo,
      DistributedReductions& reductions);
};
```

### 5.3 Distributed SCF Driver

```cpp
class DistributedSCF {
 public:
  struct Config {
    int max_iterations;
    double energy_tolerance;
    MixerType mixer;
  };

  // Run distributed SCF loop.
  // 1. Build local H (two-center + three-center + grid)
  // 2. Exchange halo H
  // 3. Solve distributed (SP2 or FOE)
  // 4. Build local rho from local P
  // 5. All-reduce rho on grid
  // 6. Compute local V_xc, V_H
  // 7. Check convergence (all-reduce energy)
  static Status Run(
      const System& system,
      const Partitioner::Result& partition,
      const Config& cfg,
      CommBackend comm = CommBackend::NCCL);
};
```

## 6. Python API Surface

```python

    basis="DZP",

    partition_method="RCB",
    truncation=1e-6,

# ASE-compatible

forces = calc.get_forces(atoms)
```

## 9. Error Handling

- All distributed operations return `Status` objects.
- Communication failures are fatal (no silent retry).
- Partition imbalance >10% triggers repartitioning.
- Halo exchange timeout → checkpoint and restart.

---

## 10. Testing Strategy

1. **Unit tests:** Partition correctness, halo exchange round-trip.
2. **Integration tests:** 2-partition SCF vs single-partition reference.
3. **Scaling tests:** Weak scaling 1→8 GPUs, measure efficiency.
4. **Regression:** Nightly distributed A/B vs FP64 reference.
